## ## v-if 和 v-show 的区别及使用场景是什么？

**区别：**

- **原理**：`v-if` 是“真正的”条件渲染，它会确保在切换过程中条件块内的事件监听器和子组件适当地被销毁和重建；`v-show` 则是简单的 CSS `display` 属性切换。

- **编译过程**：`v-if` 状态切换时会触发组件的生命周期（创建/销毁）；`v-show` 仅仅改变样式，组件始终保持在 DOM 树中。

- **性能消耗**：`v-if` 有更高的切换消耗（频繁销毁和创建）；`v-show` 有更高的初始渲染消耗（无论真假都会渲染进 DOM）。

**使用场景：**

- 如果需要频繁切换显示状态，推荐使用 `v-show`。

- 如果在运行时条件很少改变，或者初始条件为假时不希望加载资源的场景，推荐使用 `v-if`。

## ## computed 和 watch 的区别是什么？

**computed（计算属性）：**

- **缓存特性**：具有缓存性。只有当 computed 依赖的响应式数据发生变化时才会重新计算。如果依赖未变，多次访问会直接返回之前的缓存结果。

- **异步支持**：不支持异步操作。如果 computed 内部包含异步操作，将无法正常返回预期结果。

- **应用场景**：适用于多对一或者一对一的场景，主要用于由现有响应式数据派生出新数据的场景（例如：购物车总价计算、数据过滤）。

**watch（侦听器）：**

- **缓存特性**：无缓存性。每次监听的响应式数据发生变化时，都会执行设定的回调函数。

- **异步支持**：支持异步操作。可以在 watch 的回调函数中执行异步任务（例如：发起网络请求）。

- **应用场景**：适用于一对多，或者需要在数据变化时执行复杂业务逻辑或副作用操作的场景（例如：搜索框输入防抖后触发接口查询）。

## ## 为什么 Vue 中的 v-for 要求必须设置 key？

- **核心原因**：`key` 是 Vue 中 VNode（虚拟节点）的唯一标记。在进行 Virtual DOM 的 Diff 算法比较时，Vue 默认采用“就地复用”策略，key 能够打破这种默认策略。

- **准确性**：给节点提供唯一标识，帮助 Diff 算法在对比新旧节点树时准确找到对应节点，避免状态错乱（如：列表项顺序打乱时，输入框复用导致的文本错位问题）。

- **性能高效**：利用 key 可以更高效地复用已有的 DOM 节点，减少不必要的 DOM 销毁和创建过程，提升渲染性能。

- **最佳实践**：强烈建议使用数据项中唯一的标识（如 ID）作为 key，尽量避免使用数组的 `index` 作为 key。因为当数组发生插入或删除操作时，元素的 index 会发生变化，导致对应的 key 失效，从而引发大面积非必要的 DOM 更新。

## ## Vue 组件间通信方式有哪些？

Vue 组件间通信主要分为以下三种场景及对应的方法：

**1. 父子组件通信：**

- `props` / `$emit`：最基础、最常用的方式，父传子用 props，子传父触发 $emit。

- `ref` / `defineExpose`：父组件通过 ref 获取子组件实例，直接调用子组件内部暴露出来的方法或访问数据。

- `v-model`：用于实现父子组件数据的双向绑定。

**2. 跨级/祖孙组件通信：**

- `provide` / `inject`：祖先组件通过 provide 提供数据，其所有后代组件都可以通过 inject 注入并使用这些数据，无需逐层传递。

- `$attrs`：包含父作用域中不作为组件 props 被识别的 attribute 绑定。在 Vue 3 中，事件监听器（原 `$listeners`）也合并到了 `$attrs` 中，可用于透传给内部的深层组件。

**3. 兄弟组件或任意组件通信：**

- `Vuex` / `Pinia`：官方推荐的全局状态管理工具，适用于大型复杂项目。

- `EventBus`（全局事件总线）：适用于简单的小型项目。在 Vue 2 中通常使用 `new Vue()` 实例作为 EventBus；Vue 3 中移除了相关实例方法，需引入第三方库（如 `mitt`）来实现。

## ## Vue 生命周期有哪些阶段，分别有什么作用？

Vue 的生命周期是指组件从创建到销毁的整个过程，主要分为四个阶段：

**1. 创建阶段 (Creation)：**

- `beforeCreate`: 实例刚刚在内存中被创建，初始化过程刚开始，此时响应式数据（data）和方法（methods）还未挂载，无法访问。

- `created`: 实例的响应式数据、计算属性、方法和侦听器等已配置完成，但组件的模板（DOM）还未编译和挂载，无法操作真实 DOM。**（常用于发送初始的异步网络请求）**。

**2. 挂载阶段 (Mounting)：**

- `beforeMount`: 模板已经编译在内存中，但尚未挂载到页面的真实 DOM 上。

- `mounted`: 组件已经完整挂载到页面上，真实 DOM 已生成，可以通过 API 访问和操作 DOM 节点。**（常用于获取 DOM 节点、初始化第三方依赖库、绑定全局事件等）**。

**3. 更新阶段 (Updating)：**

- `beforeUpdate`: 当响应式数据发生更新时调用，此时内存中的数据是新的，但页面的真实 DOM 还是旧的，尚未重新渲染。

- `updated`: 虚拟 DOM 重新渲染并打补丁（Patch）应用到真实 DOM 后调用，此时数据和页面的 DOM 都已完成同步更新。

**4. 销毁/卸载阶段 (Destruction/Unmounting)：**

- `beforeDestroy` (Vue 3 更名为 `beforeUnmount`): 组件实例销毁前调用，此时实例仍然完全可用，数据和方法依然可以访问。**（常用于执行清理任务，如：清除定时器、解绑自定义事件或全局事件监听器，防止内存泄漏）**。

- `destroyed` (Vue 3 更名为 `unmounted`): 组件实例销毁后调用，此时该组件实例的所有指令都被解绑，所有事件监听器被移除，所有子组件也被销毁。

## ## Vue 2 和 Vue 3 的响应式原理分别是什么？

**Vue 2 响应式原理 (`Object.defineProperty`)：**

- **核心实现**：在初始化阶段，Vue 通过 `Observer` 递归遍历 data 对象的所有属性，利用原生的 `Object.defineProperty` 方法将这些属性转化为 getter 和 setter。

- **依赖收集与派发更新**：当组件 render 函数执行并访问数据时，会触发 getter，此时 `Dep`（依赖管理器）会将当前的 `Watcher`（订阅者）收集起来。当修改数据触发 setter 时，`Dep` 会通知所有收集到的 `Watcher` 执行更新，重新渲染组件。

- **存在的缺陷**：无法拦截对象属性的新增和删除操作（需借助 `$set`/`$delete`）；无法直接监听数组基于索引的修改和长度的改变（Vue 2 通过劫持/重写数组的原型方法如 push、pop 等来弥补）。

**Vue 3 响应式原理 (`Proxy`)：**

- **核心实现**：使用 ES6 的 `Proxy` 对象来代理整个状态对象，并配合 `Reflect` API 进行属性的获取和设置。

- **优势**：`Proxy` 是对整个对象进行代理，而不是具体属性，因此可以完美监听到对象属性的新增、删除，以及数组索引和长度的变化。

- **性能提升**：Vue 3 采用了**惰性代理**机制。不同于 Vue 2 在初始化时必须递归遍历深层对象，Vue 3 的 Proxy 只有在实际访问到嵌套的深层对象属性时，才会将深层对象也转化为响应式，大大减少了初始化的性能开销。

## ## 什么是 Virtual DOM (虚拟 DOM)？为什么要使用它？

**概念：** Virtual DOM 本质上是一个用原生 JavaScript 对象来描述真实 DOM 树结构的模型。它包含了诸如标签名（`tag`）、属性（`props`）、子节点（`children`）等关键信息。

**使用目的与优势：**

1. **抽象与跨平台能力**：虚拟 DOM 将 UI 的表达形式与底层的真实 DOM 渲染机制解耦。这意味着同一套 Vue 代码，通过不同的渲染器，不仅可以渲染成浏览器的真实 DOM，还可以渲染到服务器端（SSR）、移动端原生应用（如 Weex、Uni-app）等不同的目标平台。

2. **保证性能下限**：在浏览器中直接频繁操作真实 DOM 的成本非常昂贵，极易引发大面积的重排（Reflow）和重绘（Repaint）。引入虚拟 DOM 后，当状态改变时，Vue 会在内存中构建一棵新的虚拟 DOM 树，通过 Diff 算法与旧树进行比对，计算出产生变化的最小差异（Patch），最后将这些差异一次性、批量地更新到真实 DOM 上，从而有效减少了直接操作真实 DOM 的频率和范围。

## ## Vue 2 和 Vue 3 的 Diff 算法原理是什么？

Diff 算法的核心目标是比较新旧两棵 Virtual DOM 树，找出最小的差异并更新到真实 DOM，以提升渲染效率。Vue 的 Diff 算法默认只进行同层级比较，不会跨层级移动节点。

**Vue 2 的 Diff 算法 (双端比较算法)：**

- 算法在新旧节点列表的两端各设置两个指针（旧头、旧尾、新头、新尾）。

- 循环遍历时，进行四次交叉比对：旧头对比新头、旧尾对比新尾、旧头对比新尾、旧尾对比新头。

- 如果在双端比较中匹配成功，则直接复用真实 DOM 并移动相应位置。

- 如果四次比较均未匹配，则以新节点的 `key` 为基准，去旧节点的映射表中查找是否含有对应的节点。如果有则复用并移动，如果没有则创建新节点。最后处理循环结束后剩余的需要新增或删除的节点。

**Vue 3 的 Diff 算法 (最长递增子序列优化算法)：** 在借鉴了双端比较的基础上，Vue 3 进行了大幅优化：

- **前置与后置预处理**：首先从头部和尾部向中间遍历，处理掉首尾相同、不需要移动的节点。

- **求解最长递增子序列**：对于预处理后中间剩下顺序混乱的未知节点，Vue 3 会基于这些节点在新列表中的位置索引，利用算法求解出一个“最长递增子序列”。

- **最小化移动**：属于这个“最长递增子序列”中的节点，其相对顺序是保持不变的，因此**完全不需要进行 DOM 移动操作**。Diff 算法只需要对子序列之外的节点进行移动或插入，从而将 DOM 操作降到了最低限度。

## ## Vue 中 $nextTick 的作用和底层原理是什么？

**作用：** `$nextTick` 提供了一个回调函数，该回调会在 Vue 完成下一次 DOM 更新循环之后立即执行。通常用于在修改响应式数据后，需要立即获取更新后的真实 DOM 状态的场景（例如：修改数据让输入框显示后，立即调用 focus() 聚焦）。

**底层原理 (基于 Event Loop 和异步更新队列)：**

- **异步更新队列**：Vue 在侦听到数据变化时，并不会立即同步更新 DOM。而是开启一个缓冲队列，将同一事件循环（Event Loop）中所有的数据变更引发的 Watcher 更新统一收集起来，并去除重复的 Watcher，避免不必要的计算。

- **微任务优先**：`$nextTick` 的底层实现严重依赖于 JavaScript 的事件循环机制。Vue 会将 `$nextTick` 接收到的回调函数推入到一个 callbacks 数组中。然后利用原生的异步 API 将执行这些回调的任务加入到微任务（MicroTask）队列中。

- **降级策略**：Vue 内部优先尝试使用 `Promise.then` 或 `MutationObserver`（微任务）来实现延迟执行。如果浏览器不支持，则会降级使用 `setImmediate` 或 `setTimeout(fn, 0)`（宏任务）。

- **执行时机**：当同步代码执行完毕，主线程栈清空后，JavaScript 引擎会开始执行微任务队列。此时，Vue 会先执行 DOM 更新的操作，紧接着执行 `$nextTick` 队列中的回调。因此，在 `$nextTick` 回调触发时，DOM 保证已经是最新的状态。

## ## Vue 中 keep-alive 的作用和核心原理是什么？

**作用：** `<keep-alive>` 是 Vue 的内置抽象组件，主要用于包裹动态组件。它的作用是在组件切换时，将不活动的组件实例缓存在内存中，而不是销毁它们。当组件再次显示时，可以直接从缓存中恢复，从而保留组件的状态（如表单输入内容、滚动位置），并避免了重新渲染带来的性能开销。

**核心配置与生命周期：**

- 接收 `include`（正则或字符串匹配缓存）、`exclude`（匹配不缓存）和 `max`（最大缓存实例数量）三个属性。

- 被其包裹的组件会增加两个特殊的生命周期钩子：`activated`（组件被激活、从缓存中取出时调用）和 `deactivated`（组件失活、被放入缓存时调用）。

**底层原理 (LRU 缓存淘汰算法)：**

- `keep-alive` 组件内部维护了一个 `cache` 对象（以组件或 VNode 的 key 作为键，VNode 作为值进行存储）和一个 `keys` 数组（专门用于维护被缓存组件 key 的顺序）。

- 当被包裹的子组件即将挂载时，`keep-alive` 会先判断该组件是否命中缓存。如果命中，则直接取出缓存的 VNode 进行渲染，并将该组件的 key 在 `keys` 数组中的位置移动到最末尾，标记为最新访问。

- 如果配置了 `max` 属性限制最大缓存数，当存入新的组件导致缓存超载时，`keep-alive` 会采用 **LRU (Least Recently Used，最近最少使用)** 缓存淘汰策略。它会直接删除 `keys` 数组的第一个元素（即最久未被访问的节点 key），并销毁对应的组件实例和 cache 中的记录，以此腾出空间存入最新的组件。

## ## Vue 3 相比 Vue 2 做了哪些核心优化和提升？

**1. 响应式系统底层重构：** 彻底抛弃了 Vue 2 存在局限的 `Object.defineProperty`，改用 ES6 的 `Proxy` 实现响应式。支持了对对象增删属性及数组变动的完美拦截，并且采用惰性代理策略，提升了大型数据对象的初始化性能。

**2. 引入 Composition API (组合式 API)：** 新增了 `setup`、`ref`、`reactive`、`watchEffect` 等 API。解决了 Vue 2 中 Options API 在面对复杂业务时，逻辑代码按照 `data`、`methods` 等强行割裂、导致难以阅读和维护的问题，也解决了 Mixins 带来的命名冲突和数据来源不清晰的痛点，使逻辑复用更加灵活。

**3. 编译期性能优化 (Compiler)：**

- **静态提升 (Static Hoisting)**：模板中不依赖响应式数据的纯静态节点和静态属性，在编译阶段会被提升到渲染函数外部，每次更新时直接复用，避免重复创建。

- **补丁标记 (PatchFlag)**：Vue 3 编译器会为动态的节点打上特定的位运算标记（如：文本动态、Class 动态）。在 Diff 阶段，渲染器可以通过 PatchFlag 实现**靶向更新**，只对比发生了变化的属性，跳过无意义的静态比对。

**4. 更好的 Tree-Shaking 支持与包体积缩小：** Vue 3 架构上采用了更多的按需引入机制，全局 API 从直接挂载在 Vue 实例上转移到了通过 ES Module 导入。这使得构建工具（如 Webpack、Vite）能更有效地剔除未使用的代码，大幅减小了最终打包的体积。同时移除了不常用的特性如 Filters（过滤器）、内联模板等。

**5. 更好的 TypeScript 支持：** Vue 3 的源代码本身就是使用 TypeScript 进行重写的，因此其内部 API 和类型推导天然对 TS 具有极其优秀的兼容性，无需像 Vue 2 那样依赖额外的装饰器类库。

**6. 新增内置组件：** 引入了 `Fragment`（支持组件拥有多个根节点）、`Teleport`（能够将 DOM 渲染到当前组件树之外的具体位置，如用于全局弹窗）、`Suspense`（用于在异步组件加载完成前渲染后备 UI）。

## ## Vue 3 中 ref 和 reactive 的区别是什么？

在 Vue 3 的 Composition API 中，`ref` 和 `reactive` 是定义响应式状态的两个核心函数，它们的区别主要体现在适用数据类型、底层原理及使用方式上。

**适用数据类型的区别：**

- **`ref`**：主要设计用于定义**基本数据类型**（如 String、Number、Boolean）的响应式。同时，它也兼容定义复杂数据类型（对象或数组）。

- **`reactive`**：严格规定只能用于定义**复杂引用数据类型**（如 Object、Array、Map、Set 等），如果传入基本数据类型则会抛出警告且不具有响应式。

**底层原理的区别：**

- **`ref`**：本质上是返回一个带有 `.value` 属性的 `RefImpl` 类的实例对象。当我们访问或修改 `.value` 时，内部通过对象的 getter/setter 进行拦截触发响应。如果给 `ref` 传入的是一个对象，其底层其实也是自动调用了 `reactive` 来处理这个对象内部的属性。

- **`reactive`**：底层直接依赖 ES6 的 `Proxy` 对象来实现深度代理，直接拦截对象本身的各种操作。

**使用方式及注意事项的区别：**

- **`ref`**：在 JavaScript/TypeScript 逻辑代码中读取或赋值时，**必须加上 `.value`**（例如：`count.value++`）。但在 Vue 模板（`<template>`）中直接使用时，Vue 提供了自动解包功能，不需要加 `.value`。

- **`reactive`**：在任何地方使用都不需要加 `.value`，像普通 JavaScript 对象一样直接访问属性即可。

- **解构限制**：将 `reactive` 对象的属性进行 ES6 结构（例如：`const { name } = reactive({ name: 'Vue' })`），或者将整个 reactive 变量重新赋值为一个新对象，都会导致这些数据**丢失响应式关联**。如需解构并保持响应式，必须借助 `toRefs` 或 `toRef` 函数处理。

## ## Vue 3.4+ 中 `defineModel` 的作用是什么？

**核心作用：** `defineModel` 是一个宏，旨在简化组件间双向绑定（`v-model`）的实现。在 Vue 3.4 之前，实现自定义 `v-model` 需要声明 `props` 和 `emits`，并手动监听变化进行抛出。

**优势：**

- **简化代码**：通过 `const model = defineModel()` 一行代码即可获取一个可直接读写的响应式对象。
- **自动同步**：对 `model.value` 的任何修改都会自动触发底层 `update:modelValue` 事件，无需手动 `$emit`。

## ## React Fiber 架构解决了什么问题？其实现原理是什么？

**解决的核心问题：** 在 React 15 及以前，Stack Reconciler 采用递归遍历虚拟 DOM 树，这个过程是**同步、不可中断的**。如果 DOM 树过于庞大，JS 主线程会长时间被占用，导致浏览器无法处理高优先级的任务（如动画、点击响应），界面会出现卡顿感。

**核心实现原理：**

1. **任务拆分（Incremental Rendering）**：将更新过程拆分为多个微小的“工作单元（Fiber）”，每个 Fiber 对应一个 React 元素。
2. **异步可中断（Concurrency）**：引入了调度器（Scheduler）。在执行更新时，会检查当前帧的剩余时间（RequestIdleCallback 思想）。如果时间不够，React 会保存当前执行进度，将控制权交还给浏览器，等待下一帧空闲时继续。
3. **双缓存技术**：内存中同时存在两棵 Fiber 树（Current Tree 和 WorkInProgress Tree），通过指针切换实现平滑更新。
4. **优先级调度**：根据任务类型（如动画 > 用户交互 > 数据加载）分配不同的优先级。

## ## React 中 `useEffect` 和 `useLayoutEffect` 的区别是什么？

**执行时机：**

- **`useEffect`**：在浏览器渲染完成（即屏幕已更新）之后**异步**执行。它不会阻塞浏览器的绘图过程，是处理大多数副作用（如 API 请求、订阅）的首选。
- **`useLayoutEffect`**：在 DOM 更新之后、浏览器绘制（Paint）之前**同步**执行。它会阻塞浏览器，直到回调执行完毕。

**适用场景：**

- 绝大多数情况使用 `useEffect`。
- 如果副作用需要修改 DOM 且引起布局变化（如测量元素尺寸、防止闪烁），必须使用 `useLayoutEffect` 以保证用户看不到中间状态。

## ## React 中 `useMemo` 和 `useCallback` 的区别是什么？

**共同点：** 两者都是为了进行性能优化，通过依赖数组来缓存结果，避免不必要的重复计算或组件重渲染。

**主要区别：**

- **`useMemo`**：缓存的是**函数的返回值**（计算结果）。
- **`useCallback`**：缓存的是**函数实例本身**。它等价于 `useMemo(() => fn, [deps])`。

**使用目的：**

- `useMemo`：用于避免复杂计算在每次渲染时重复执行。
- `useCallback`：用于将函数作为 props 传递给子组件（配合 `React.memo` 使用）时，保持函数引用不变，防止子组件无谓地重渲染。

## ## React 和 Vue 的核心区别（选型对比）有哪些？

**1. 编程范式：**

- **React**：主张 **"All in JS"** (JSX)。认为 UI 是状态的纯函数。灵活性极高，生态及其庞大。
- **Vue**：主张 **"SFC (单文件组件)"** 和模板。通过指令将数据和逻辑解耦。学习曲线相对平缓。

**2. 响应式原理：**

- **React**：基于**不可变数据（Immutability）**和手动触发更新（`setState`）。当状态改变时，默认会重新执行整个组件及其子树的 render。
- **Vue**：基于**依赖追踪（Dependency Tracking）**。自动收集依赖，数据变化时能实现“靶向更新”，精准刷新受影响的组件节点。

**3. 语法风格：**

- **React**：更偏向原生 JS 逻辑，强调逻辑复用（Hooks 和高阶组件）。
- **Vue**：提供了大量的语法糖（指令、计算属性），开发效率较高，配置化更强。

## ## 从输入 URL 到页面展示，浏览器中间做了哪些事？

**核心流程：** 这是一个涉及网络、渲染引擎和 JS 引擎的复杂过程，主要分为以下阶段：

1. **网络请求阶段**：进行 DNS 解析、TCP 三次握手、发送 HTTP 请求。
2. **解析阶段**：
   - **HTML 解析**：浏览器将 HTML 字节流解析为 **DOM 树**。
   - **CSS 解析**：解析 CSS 文件生成 **CSSOM 树**。
3. **渲染流水线**：
   - **生成渲染树 (Render Tree)**：合并 DOM 和 CSSOM，过滤掉不可见元素（如 `display:none`）。
   - **布局 (Layout)**：计算每个节点在屏幕上的确切位置和大小。
   - **分层 (Layering)**：为复杂的层（如 3D 转换、z-index）生成单独的图层。
   - **绘制 (Painting)**：生成绘制指令。
   - **合成 (Compositing)**：将图层分为图块，通过 GPU 最终合成为屏幕影像。

## ## 浏览器 HTTP 缓存（强缓存与协商缓存）的原理是什么？

**1. 强缓存 (Strong Cache)：** 浏览器直接从本地缓存读取资源，不发送 HTTP 请求。

- **Expires**：HTTP/1.0 字段，记录绝对过期时间（受本地时间影响）。
- **Cache-Control**：HTTP/1.1 字段，优先级更高。常用值如 `max-age=3600`（相对时间）。

**2. 协商缓存 (Negotiation Cache)：** 浏览器向服务器询问资源是否过期。如果未过期，服务器返回 **304 Not Modified**。

- **Last-Modified / If-Modified-Since**：基于文件最后修改时间（精度为秒）。
- **Etag / If-None-Match**：基于文件内容哈希值，优先级更高，解决时间精度和文件未变但时间变了的问题。

## ## 什么是 JavaScript 的事件循环 (Event Loop)？宏任务与微任务的区别？

**核心原理：** 由于 JS 是单线程的，通过事件循环机制实现非阻塞异步操作。

- **执行顺序**：同步代码（执行栈） -> **微任务队列**（一次清空） -> 渲染 -> **宏任务队列**（取一个执行）。
- **宏任务 (Macrotask)**：`setTimeout`, `setInterval`, `setImmediate`, I/O, UI 渲染。
- **微任务 (Microtask)**：`Promise.then/catch`, `Async/Await` (实质是 Promise), `process.nextTick` (Node.js)。

**记忆技巧**：每次宏任务执行完，都会先检查并清空所有的微任务。
