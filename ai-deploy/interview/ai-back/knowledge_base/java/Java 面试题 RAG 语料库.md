## ## Java 中 `==` 和 `equals()` 的区别是什么？

**核心区别：**

- **`==` 运算符**：
  
  - 对于**基本数据类型**（如 int, double, boolean 等），`==` 比较的是它们的**值**是否相等。
  
  - 对于**引用数据类型**（如对象、数组），`==` 比较的是它们在内存中的**对象地址**是否相同，即判断它们是否指向堆内存中的同一个对象。

- **`equals()` 方法**：
  
  - `equals()` 是 `Object` 类的方法。在 `Object` 类的默认实现中，`equals()` 的行为与 `==` 完全相同，也是比较对象的内存地址。
  
  - 但是，Java 中的大多数核心类（如 `String`、`Integer`、`Date` 等）都**重写（Override）**了 `equals()` 方法。重写后的 `equals()` 方法通常用于比较两个对象的**内容（逻辑值）**是否相等，而不是比较内存地址。

**补充说明：** 如果在自定义类中需要比较对象的逻辑状态是否相同，必须重写 `equals()` 方法。同时，根据 Java 规范，重写 `equals()` 的同时必须重写 `hashCode()` 方法，以保证在哈希表（如 HashMap、HashSet）中能正确工作。

## ## Java 中 `String`、`StringBuffer` 和 `StringBuilder` 的区别与使用场景分别是什么？

**区别：**

- **`String`**：
  
  - **不可变性**：底层使用 `final` 修饰的字符数组（JDK 8 及以前是 `char[]`，JDK 9 之后是 `byte[]`）存储。一旦创建，其值就不能被修改。
  
  - **线程安全**：由于不可变，天然是线程安全的。
  
  - **性能**：每次对 `String` 进行拼接或修改操作，都会在内存中生成一个新的 `String` 对象，频繁修改会导致产生大量废弃对象，影响性能（即使编译器有时会优化为 StringBuilder）。

- **`StringBuffer`**：
  
  - **可变性**：继承自 `AbstractStringBuilder`，底层也是字符数组但没有 `final` 修饰，内容可以被修改。
  
  - **线程安全**：是线程安全的。它的绝大多数公开方法（如 `append`、`insert`）都使用了 `synchronized` 关键字修饰，保证了多线程环境下的同步。
  
  - **性能**：由于加了同步锁，性能略低于 `StringBuilder`。

- **`StringBuilder`**：
  
  - **可变性**：与 `StringBuffer` 类似，内容可变。
  
  - **线程安全**：**非线程安全**。方法没有同步锁。
  
  - **性能**：在单线程环境下，性能是三者中最高的。

**使用场景：**

- 操作少量字符串数据，或者字符串初始化后不再改变，使用 `String`。

- 在**单线程**环境下进行频繁的字符串拼接或修改操作，推荐使用 `StringBuilder`。

- 在**多线程**环境下进行频繁的字符串拼接或修改操作，必须使用 `StringBuffer`。

## ## Java 中 `HashMap` 的底层实现原理（JDK 1.8）是什么？

**底层数据结构：** 在 JDK 1.8 中，`HashMap` 的底层数据结构是由 **数组 + 链表 + 红黑树** 组成的。

**核心工作机制：**

- **哈希计算与定位**：当向 `HashMap` 中 `put(key, value)` 时，首先调用 key 的 `hashCode()` 方法计算出哈希值，然后经过一次扰动函数（高低位异或）降低哈希冲突概率，最后通过 `(n - 1) & hash`（n 为数组长度）计算出该元素在数组中的索引位置。

- **处理哈希冲突**：
  
  - 如果定位到的数组位置为空，直接将节点放入。
  
  - 如果位置不为空（发生哈希冲突），会检查 key 是否相同。如果相同则覆盖 value。
  
  - 如果 key 不同，则将新节点以**尾插法**（JDK 1.8 之前是头插法，容易在扩容时产生死循环）挂载到该位置的链表末尾。

- **链表转红黑树（树化机制）**：当链表的长度大于等于 **8**，**并且**整个 HashMap 的数组长度大于等于 **64** 时，该链表会转换为红黑树。这大大将查询时间复杂度从 O(n) 降低到了 O(log n)。如果数组长度小于 64，优先选择数组扩容而不是树化。

- **扩容机制（Resize）**：当 HashMap 中的元素个数超过了阈值（`threshold = capacity * loadFactor`，默认负载因子为 0.75）时，会触发扩容。数组容量会扩大为原来的 **2 倍**。扩容时，JDK 1.8 进行了优化，利用二进制特性，节点要么留在原索引位置，要么移动到“原索引 + 旧容量”的位置，避免了重新计算所有 hash 的开销。

## ## Java 并发编程中 `synchronized` 和 `ReentrantLock` 的区别是什么？

**核心区别：**

- **底层实现：**
  
  - `synchronized` 是 Java 中的**关键字**，属于 JVM 层面实现的隐式锁。底层通过 Monitor 对象实现（包含 `monitorenter` 和 `monitorexit` 指令）。
  
  - `ReentrantLock` 是 `java.util.concurrent.locks` 包下的一个**类**，属于 API 层面的显式锁。底层基于 AQS（AbstractQueuedSynchronizer）和 CAS（Compare-And-Swap）操作实现。

- **加锁与释放锁的方式：**
  
  - `synchronized` 是自动加锁和释放锁的。发生异常时，JVM 会自动释放线程占有的锁，不会导致死锁。
  
  - `ReentrantLock` 必须手动调用 `lock()` 方法加锁，并且**必须在 `finally` 块中手动调用 `unlock()`** 方法释放锁，否则如果发生异常可能会导致死锁。

- **锁的灵活性与高级特性：**
  
  - `synchronized` 只能是非公平锁。
  
  - `ReentrantLock` 功能更强大。它可以指定为**公平锁或非公平锁**（构造函数传入 true 为公平锁）。它还提供了可响应中断的获取锁（`lockInterruptibly`）、可超时的获取锁（`tryLock`）等机制。

- **条件变量（Condition）：**
  
  - `synchronized` 只能结合 `Object` 类的 `wait()`/`notify()` 实现单一的等待通知机制。
  
  - `ReentrantLock` 可以结合多个 `Condition` 实例实现多路通知机制，可以精确唤醒指定的线程。

## ## Java 线程池 `ThreadPoolExecutor` 的核心参数有哪些？其工作原理是什么？

**七个核心参数：**

1. **`corePoolSize`（核心线程数）**：线程池中常驻的线程数量，即使它们处于空闲状态也不会被销毁（除非设置了 `allowCoreThreadTimeOut`）。

2. **`maximumPoolSize`（最大线程数）**：线程池能够容纳同时执行的最大线程数。

3. **`keepAliveTime`（空闲线程存活时间）**：当线程池中的线程数量超过 `corePoolSize` 时，多余的空闲线程在被销毁前等待新任务的最长时间。

4. **`unit`（时间单位）**：`keepAliveTime` 的时间单位。

5. **`workQueue`（任务队列）**：用于缓存处于等待执行状态的任务的阻塞队列（如 `ArrayBlockingQueue`, `LinkedBlockingQueue`）。

6. **`threadFactory`（线程工厂）**：用于创建新线程的工厂，可以给线程设置有意义的名字，方便排查问题。

7. **`handler`（拒绝策略）**：当任务队列已满，且线程池中的线程数量已达到 `maximumPoolSize` 时，对待新加入任务的处理策略（如抛出异常 `AbortPolicy`、由调用者线程执行 `CallerRunsPolicy` 等）。

**工作原理（任务提交流程）：**

1. 提交新任务时，如果当前正在运行的线程数小于 `corePoolSize`，则立刻创建新的核心线程执行该任务。

2. 如果当前运行的线程数大于等于 `corePoolSize`，则将任务放入 `workQueue`（阻塞队列）中排队等待。

3. 如果 `workQueue` 已经满了，且当前运行的线程数小于 `maximumPoolSize`，则创建新的非核心线程去执行任务。

4. 如果 `workQueue` 已满，且运行的线程数达到了 `maximumPoolSize`，线程池将触发指定的**拒绝策略**（`handler`）来处理该任务。

## ## Java 并发编程中 `volatile` 关键字的作用和底层原理是什么？

**两大核心作用：**

1. **保证变量的内存可见性**：当一个线程修改了被 `volatile` 修饰的变量的值，新值会立刻被强制刷新到主内存中；当其他线程读取该变量时，会强制使它们工作内存中的缓存失效，直接从主内存中读取最新值。这就保证了多线程环境下对该变量修改的可见性。

2. **禁止指令重排序**：编译器和处理器为了优化性能，常常会对指令进行重排序。`volatile` 关键字可以通过插入“内存屏障”（Memory Barrier）来禁止编译器和底层的 CPU 对修饰的变量进行重排序优化，从而保证代码执行的有序性（常用于单例模式的双重检查锁 DCL 中）。

**不保证原子性：** `volatile` 虽然能保证可见性和有序性，但**不能保证复合操作的原子性**（例如 `i++` 操作，它实际上包含了读取、加一、写回三个步骤）。如果需要保证原子性，需要使用 `synchronized`、`ReentrantLock` 或 JUC 包下的原子类（如 `AtomicInteger`）。

**底层原理：** `volatile` 的底层是通过 CPU 的 **Lock 指令**（在 x86 架构下）和**内存屏障**来实现的。Lock 指令会触发硬件级别的缓存一致性协议（如 MESI 协议），确保多个 CPU 核心之间的高速缓存数据保持一致。

## ## JVM 的运行时数据区（内存结构）包含哪些部分？分别有什么作用？

根据 Java 虚拟机规范，JVM 在执行 Java 程序时会把它所管理的内存划分为若干个不同的数据区域，主要分为**线程私有**和**线程共享**两大类：

**线程私有区域（生命周期与线程相同）：**

1. **程序计数器（Program Counter Register）**：是一块较小的内存空间，保存着当前线程所执行的字节码指令的行号指示器。它是唯一一个在 JVM 规范中没有规定任何 `OutOfMemoryError` 情况的区域。

2. **Java 虚拟机栈（VM Stack）**：描述的是 Java 方法执行的内存模型。每个方法在执行时都会同步创建一个“栈帧”用于存储局部变量表、操作数栈、动态链接、方法出口等信息。方法调用到结束对应着栈帧的入栈和出栈。

3. **本地方法栈（Native Method Stack）**：与虚拟机栈的作用类似，区别在于虚拟机栈为执行 Java 方法服务，而本地方法栈为 JVM 使用到的 Native 方法（通常由 C/C++ 编写）服务。

**线程共享区域（所有线程可访问）：** 4. **Java 堆（Heap）**：是 JVM 所管理的内存中最大的一块。几乎所有的**对象实例和数组**都在这里分配内存。这里也是垃圾收集器（GC）管理的主要区域，因此常被称为“GC 堆”。
5. **方法区（Method Area）**：用于存储已被 JVM 加载的**类信息、常量、静态变量、即时编译器编译后的代码缓存**等数据。在 JDK 1.8 之前，HotSpot 虚拟机用“永久代”实现方法区；在 JDK 1.8 之后，废弃了永久代，改用在本地内存中实现的“元空间（MetaSpace）”来替代。

## ## JVM 的垃圾回收（GC）机制中，如何判断一个对象是否可以被回收？

在 Java 中，垃圾收集器在对堆内存进行回收前，首先要确定哪些对象是“存活”的，哪些是“死亡”（可以被回收）的。JVM 主要采用**可达性分析算法**来进行判断。

**1. 引用计数法（早期/非主流）：**

- **原理**：在对象中添加一个引用计数器，每当有一个地方引用它，计数器加 1；引用失效时减 1。计数器为 0 的对象即可被回收。

- **缺点**：Java 虚拟机**没有采用**这种方法，因为它很难解决对象之间的**循环引用**问题（如对象 A 引用 B，B 又引用 A，即使它们都不再被外界使用，计数器也不为 0，导致内存泄漏）。

**2. 可达性分析算法（Reachability Analysis - JVM 当前采用）：**

- **原理**：通过一系列被称为 **"GC Roots"** 的根对象作为起始节点集，从这些节点开始，根据引用关系向下搜索，搜索过程所走过的路径称为“引用链”。如果某个对象到 GC Roots 间没有任何引用链相连（即 GC Roots 到该对象不可达），则证明该对象是不可用的，可以被判定为可回收对象。

- **常见的 GC Roots 对象包括**：
  
  - 虚拟机栈（栈帧中的局部变量表）中引用的对象。
  
  - 本地方法栈中 JNI（Native 方法）引用的对象。
  
  - 方法区中类静态属性引用的对象。
  
  - 方法区中常量引用的对象。
  
  - 所有被同步锁（synchronized）持有的对象。

## ## JVM 常见的垃圾回收算法有哪些？

JVM 主要采用分代收集理论，针对不同年代的对象特性使用不同的垃圾回收算法。常见的基础算法有以下三种：

**1. 标记-清除算法（Mark-Sweep）：**

- **过程**：分为“标记”和“清除”两个阶段。首先通过可达性分析标记出所有需要回收的对象，标记完成后，统一清除掉所有被标记的对象。

- **缺点**：执行效率不稳定；最大的问题是**会产生大量不连续的内存碎片**。如果未来需要分配大对象，可能会因为找不到足够的连续内存而提前触发下一次 GC。

**2. 标记-复制算法（Mark-Copy）：**

- **过程**：将可用内存按容量划分为大小相等的两块，每次只使用其中一块。当这一块内存用完了，就将还存活着的对象复制到另一块上面，然后再把已使用过的内存空间一次清理掉。

- **优点**：实现简单，运行高效，**不会产生内存碎片**。

- **缺点**：将可用内存缩小为了原来的一半，空间浪费较大。

- **应用场景**：现代 JVM 的**新生代**广泛采用这种算法（新生代中绝大多数对象朝生夕灭，所以不需要 1:1 划分，而是分为较大的 Eden 区和两个较小的 Survivor 区）。

**3. 标记-整理算法（Mark-Compact）：**

- **过程**：标记过程与“标记-清除”算法一样。但后续步骤不是直接清理可回收对象，而是让所有存活的对象都向内存空间的一端移动，然后直接清理掉边界以外的内存。

- **优点**：解决了内存碎片问题，同时也避免了复制算法中内存减半的代价。

- **缺点**：移动存活对象需要暂停用户线程（即发生 "Stop The World"，STW），在对象存活率高的情况下移动成本很大。

- **应用场景**：JVM 的**老年代**通常采用此算法（因为老年代对象存活率高，不适合复制算法）。

## ## Java 的类加载机制和双亲委派模型是什么？有什么作用？

**类加载机制：** JVM 把描述类的数据从 Class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被 JVM 直接使用的 Java 类型，这就是虚拟机的类加载机制。整个生命周期包括：加载（Loading）、验证（Verification）、准备（Preparation）、解析（Resolution）、初始化（Initialization）、使用和卸载。

**双亲委派模型（Parents Delegation Model）：**

- **定义**：Java 虚拟机提供了不同的类加载器（如启动类加载器 Bootstrap、扩展类加载器 Extension、应用程序类加载器 Application）。双亲委派模型要求：除了顶层的启动类加载器外，其余的类加载器都应当有自己的父类加载器。当一个类加载器收到类加载的请求时，它首先**不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成**，每一个层次的类加载器都是如此，最终所有的加载请求都会传送到最顶层的启动类加载器中。只有当父加载器反馈自己无法完成这个加载请求（它的搜索范围中没有找到所需的类）时，子加载器才会尝试自己去完成加载。

- **核心作用（为什么要有双亲委派）：**
  
  1. **保证核心类库的安全性**：确保 Java 核心类库（如 `java.lang.Object`、`java.lang.String`）不被恶意篡改或替换。即使黑客自己编写了一个 `java.lang.String` 类，按照双亲委派机制，最终也会委派给最顶层的 Bootstrap ClassLoader 加载 rt.jar 中的标准 String 类，而不会加载自定义的恶意类。
  
  2. **避免类的重复加载**：当父加载器已经加载过某一个类时，子加载器就不会再重新加载，保证了在 JVM 内存中，同一个全限定名的类只存在一个 Class 对象实例。

## ## JDK 21 中的虚拟线程（Virtual Threads）是什么？解决了什么问题？

**核心概念：** 虚拟线程是 JDK 21 正式引入的一种轻量级线程（属于 Project Loom）。它不再与操作系统的内核线程（Kernel Thread）一一对应，而是由 JVM 在用户态进行管理和调度。

**解决的核心问题：**

- **极高的并发能力**：传统的平台线程（Platform Thread）非常昂贵（默认占 1MB 栈空间），一台机器支撑数千个线程就已达到极限。虚拟线程极其廉价，可以在单个 JVM 中创建**百万级别**的实例。
- **吞吐量提升**：针对 I/O 密集型任务，当虚拟线程执行阻塞操作（如请求数据库）时，JVM 会自动将其从内核线程上“卸载”并挂起，让出内核线程给其他虚拟线程使用。等 I/O 完成后，JVM 再将其“挂载”回来继续执行。

**与线程池的区别：** 虚拟线程是为了消耗更少的资源而设计的，它通常不建议使用线程池来管理。对于每一个任务，我们直接创建一个新的虚拟线程即可（`Thread.ofVirtual().start(task)`）。

## ## JVM 中 G1 垃圾收集器的工作原理是什么？

**核心理念：** G1 (Garbage First) 是一款面向服务端应用的垃圾收集器，旨在实现**高吞吐量的同时，尽可能缩短并控制停顿时间 (STW)**。

**内存布局变化：** G1 彻底放弃了传统的固定堆内存分代划分。它将整个 Java 堆划分为约 2048 个大小相等的独立区域（**Region**）。每个 Region 都可以根据需要扮演 Eden、Survivor 或 Old 角色。

**核心工作流程：**

1. **初始标记 (Initial Marking)**：标记 GC Roots 能直接关联到的对象，需要 STW，但耗时极短。
2. **并发标记 (Concurrent Marking)**：从根对象开始对堆中对象进行可达性分析，找出存活对象。此阶段与用户线程并发执行。
3. **最终标记 (Final Marking)**：处理并发标记阶段遗留的少量变动，需要 STW。
4. **筛选回收 (Live Data Counting and Evacuation)**：计算各个 Region 的回收价值和成本，根据用户期望的停顿时间制定回收计划。G1 会优先回收价值最大的 Region（这就是 Garbage First 的由来），采用**标记-复制算法**将存活对象移到空 Region 中。

**优势：** 可预测的停顿时间模型；通过 Region 划分解决了空间碎片问题。适用于 6GB 以上的大内存堆。

## ## Java 中的强、软、弱、虚四种引用分别是什么？

**1. 强引用 (Strong Reference)：** 最普遍的引用（如 `Object obj = new Object()`）。只要强引用还在，垃圾收集器**永远不会**回收掉被引用的对象。

**2. 软引用 (Soft Reference)：** 描述有用但非必须的对象。在系统**准备发生内存溢出（OOM）之前**，垃圾收集器会将这些对象列入回收范围进行二次回收。如果回收后内存仍不足，才会抛出 OOM。常用于实现**内存敏感的缓存**。

**3. 弱引用 (Weak Reference)：** 强度比软引用更弱。无论当前内存是否足够，只要**发生垃圾回收**，弱引用关联的对象都会被回收。常用于 `WeakHashMap` 或 `ThreadLocal` 内部。

**4. 虚引用 (Phantom Reference)：** 最弱的引用。无法通过虚引用获取对象实例。它的唯一作用是对象在被回收时能收到一个**系统通知**（关联引用队列 ReferenceQueue 使用），主要用于跟踪对象被回收的状态。

## ## MySQL 的 InnoDB 引擎为什么要使用 B+ 树作为索引结构？

**核心原理：** 索引的本质是减少磁盘 I/O 次数。相比于其他数据结构，B+ 树具有以下显著优势：

- **更高的阶数（更矮胖）**：B+ 树的非叶子节点只存储索引（Key），不存储数据。这使得每个节点能容纳更多的索引项，导致树的高度极低（通常 3-4 层即可支撑千万级数据），从而将磁盘 I/O 降到最低。
- **磁盘预读优化**：B+ 树节点的大小通常设置为磁盘页的倍数（默认 16KB）。
- **范围查询性能卓越**：B+ 树的所有数据都存储在叶子节点，并且叶子节点之间通过**双向链表**连接。这使得进行范围查找（如 `WHERE val > 10`）时，只需定位到起始点，然后顺着链表遍历即可。
- **查询效率稳定**：任何查询都必须从根节点走到叶子节点，路径长度相同，查询时间复杂度稳定为 O(log n)。

## ## MySQL 的事务隔离级别有哪些？MVCC 是如何实现可重复读（RR）的？

**四种事务隔离级别：** 1. 读未提交 (Read Uncommitted)、2. 读已提交 (Read Committed)、3. 可重复读 (Repeatable Read - MySQL 默认)、4. 串行化 (Serializable)。

**MVCC (多版本并发控制) 实现原理：**

- **隐式字段**：InnoDB 为每行记录添加了三个隐式字段：`DB_TRX_ID`（最后修改该行的事务 ID）、`DB_ROLL_PTR`（回滚指针，指向 Undo Log）、`DB_ROW_ID`（隐藏主键）。
- **Undo Log (回滚日志)**：记录了数据的历史版本。通过回滚指针，可以将不同版本的 Undo Log 串联成一条**版本链**。
- **ReadView (一致性视图)**：这是 MVCC 实现的关键。
  - 在 **RC** 级别：事务中**每一次**执行普通 SELECT 语句时都会生成一个新的 ReadView，因此能读到其他事务已提交的修改。
  - 在 **RR** 级别：事务中**只有第一次**执行普通 SELECT 语句时会生成 ReadView，后续查询均复用该视图。通过对比版本链中的事务 ID 与 ReadView 中的活跃事务列表，确保事务能一直读取到“初次见到的那个版本”，从而实现可重复读。

## ## Redis 持久化机制 RDB 和 AOF 的区别及优缺点是什么？

**1. RDB (Redis Database) - 快照持久化：**

- **原理**：在指定的时间间隔内，将内存中的数据生成快照存入磁盘文件（`dump.rdb`）。
- **优点**：文件紧凑，适合备份和灾难恢复；恢复大数据集时的速度远快于 AOF。
- **缺点**：实时性差。如果服务器宕机，会丢失最后一次快照后的所有数据。bgsave 过程 fork 子线程在处理超大内存时可能导致瞬间停顿。

**2. AOF (Append Only File) - 日志持久化：**

- **原理**：将每一条写操作指令记录在日志文件（`appendonly.aof`）中。
- **优点**：数据安全性高。支持每秒同步一次（Everysec），最多丢失一秒数据。
- **缺点**：文件体积比 RDB 大；恢复速度慢。

**3. 混合持久化 (JDK 4.0+ 推荐)**：开启 `aof-use-rdb-preamble`。RDB 存储存量数据，AOF 记录增量指令，兼顾了两者的长处。

## ## Redis 分布式锁在实现时需要注意哪些核心要素？

**核心要素（以 Redisson/Lua 脚本为例）：**

- **互斥性**：使用 `SETNX` (SET if Not eXists) 指令确保只有一个请求能获取锁。
- **超时自动释放**：必须设置 `EXPIRE` 过期时间，防止由于某个客户端宕机导致“死锁”。
- **原子性操作**：获取锁和设置过期时间必须是原子的，或者使用 Lua 脚本执行（防止获取锁后没来得及设过期时间就挂了）。
- **续期机制 (Watchdog)**：如果业务执行过长超过了锁时间，需要自动延长过期时间，防止业务没执行完锁就开了。
- **解铃还须系铃人**：释放锁时必须校验身份（如使用随机 UUID），防止 A 线程解了 B 线程的锁。

## ## 分布式系统的 CAP 理论和 BASE 理论分别指什么？

**1. CAP 理论：**

- **C (Consistency - 一致性)**：所有节点在同一时刻都能读取到最新的数据。
- **A (Availability - 可用性)**：系统保证每个请求都能收到响应（不保证是最新的）。
- **P (Partition Tolerance - 分区容错性)**：即使由于网络故障导致节点间无法通信，系统仍能继续运行。
- **结论**：在分布式系统中，P 是必须保证的，因此系统通常在 **CP**（一致性优先）或 **AP**（可用性优先）之间权衡。

**2. BASE 理论（对 CAP 中一致性和可用性的折中）：**

- **BA (Basically Available - 基本可用)**：允许损失部分可用性，如响应时间增加或服务降级。
- **S (Soft State - 软状态)**：允许系统存在中间状态，即数据同步存在延迟。
- **E (Eventually Consistent - 最终一致性)**：不保证实时一致，但经过一段时间后，数据必须达到一致状态。

## ## 微服务中 Nacos 注册中心的工作流程与心跳机制是什么？

**核心流程：**

1. **服务注册**：服务提供者启动时，通过 REST 请求向 Nacos Server 注册自己的 IP、端口、服务名等元数据。
2. **服务心跳**：提供者默认每隔 **5 秒** 发送一次心跳（Beat）包。如果 Nacos 超过 **15 秒** 没收到心跳，会将实例标记为不健康；超过 **30 秒** 则剔除实例。
3. **服务发现（获取列表）**：服务消费者通过 Nacos Server 获取健康实例列表。为了提升性能，消费者通常会**本地缓存**一份列表并订阅变更通知（UDP 指推结合模式）。
4. **服务健康检查**：Nacos Server 会主动检查实例的可达性。
